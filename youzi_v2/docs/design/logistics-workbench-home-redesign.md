# 物流工作台首页改版设计

> **落地状态（2026-07-18）**：首版已落地 **阶段一～三**（紧凑页头、今日重点、待处理事项+近期到港、运输概览、`GET /api/v1/workbench/overview`、模块容错与深链）。阶段四（报价跟进、快递异常、个人化、趋势）与「数据待修正」入口本轮未做。

## 背景

当前首页同时承担了工作台总览、快捷导航、海运统计、异常摘要、待办提醒和消息提醒等多种职责。同一批数据又可能同时出现在侧边栏、顶部铃铛、首页摘要和首页明细中，导致信息重复、首屏重点不明确。

本次改版将首页定位为日常操作入口：

```text
今天需要处理什么，未来几天将发生什么。
```

首页负责摘要、优先级和跳转，不承载各业务模块的完整处理流程。

## 改版目标

1. 用户进入首页后，能够立即看到最需要处理的事项。
2. 首页首屏只保留稳定、可解释、可点击的核心指标。
3. 异常、催款、轨迹审批和近期到港使用统一的信息层级。
4. 首页、顶部铃铛、侧边栏和业务页面共用同一套状态口径。
5. 减少重复入口、重复数字和大段明细列表。
6. 单个数据接口失败时，不影响其他首页模块使用。
7. 为报价跟进、快递轨迹异常等后续事项保留扩展位置。

## 非目标

第一版不包含：

- 用户自定义首页模块和排序。
- 在首页完成复杂异常处理、审批或催款流程。
- 真实历史趋势图和同比、环比分析。
- 全量运单、全量待办或全部海运节点展示。
- 按岗位配置不同版本首页。

以上能力后续可以扩展，但不应增加第一版复杂度。

## 当前主要问题

### 信息职责混杂

当前首页同时存在：

- 大标题与头部汇总数字
- 快捷入口
- 七张海运状态统计卡
- 异常跟踪摘要
- 批次待办和消息提醒
- 海运预警与 ETA 提醒
- 关注运单
- 订阅港预警
- 船名航次数据缺失提示

这些内容同时出现在一个页面层级中，用户难以判断应该优先查看统计、提醒还是业务列表。

### 数字重复

异常、审批、运单等数量可能同时出现在：

```text
侧边栏徽标
顶部待办铃铛
首页头部汇总
首页统计卡
首页明细列表
```

首页改版后只保留对当天工作有帮助的核心数字，其余数据进入对应业务页面。

### 行动数据与规模数据混排

例如“查验中”“严重超时”需要人工关注，而“在途”“已到港”主要用于了解规模。两类数据不应使用完全相同的视觉权重。

### 静态装饰趋势

当前统计卡上的小折线不来自真实历史数据，容易被理解为趋势。改版后移除装饰折线；只有接入真实时间序列后才展示趋势。

### 历史脏数据占据首页

长期未更新、明显过期但仍标记在途的运单，不应作为普通关注运单长期占据首页。这类记录应归入独立的“数据待修正”或“长期无更新”入口。

## 页面信息架构

首页从上到下分为四层：

```text
紧凑页头
今日重点
待处理事项 + 近期到港
运输概览
```

推荐桌面布局：

```text
物流工作台                                      最后更新 16:30  [刷新]

今日重点
┌────────────┬────────────┬────────────┬────────────┐
│ 待处理异常 │ 待催收款项 │ 轨迹审批   │ 三天内到港 │
│     89     │    99+     │     66     │     13     │
└────────────┴────────────┴────────────┴────────────┘

┌──────────────────────────────┬──────────────────────┐
│ 待处理事项                    │ 近期到港             │
│ 全部 / 异常 / 催款 / 审批     │ 今天 / 明天 / 后天以后│
│ 默认显示前 6～8 条            │ 默认显示前 5～6 条   │
└──────────────────────────────┴──────────────────────┘

运输概览
在途 | 查验中 | 已到港 | 本周签收
```

桌面端主内容建议为约 `2/3 + 1/3` 两列。移动端按“今日重点、待处理事项、近期到港、运输概览”的顺序纵向排列。

## 紧凑页头

页头只保留：

- 页面名称：物流工作台
- 最后成功更新时间
- 手动刷新图标按钮

移除：

- `Operations` 英文眉题
- 大面积 Hero 容器
- “需关注、海运提醒、批次待办”等重复汇总
- 船期监控、运单列表文字按钮

业务导航已经由侧边栏承担，首页不再重复提供快捷导航。

## 今日重点

今日重点固定展示四项，即使数量为 `0` 也保留位置，避免页面每天发生布局变化。

### 待处理异常

显示文案：`待处理异常`

统计口径：

```text
当前状态为待处理或处理中
且尚未关闭、解决或取消的运单异常
```

不包含历史已处理异常。

点击跳转：

```text
/shipment-exceptions?status=open
```

视觉建议：红色，仅用于数字、图标和轻量强调，不使用大面积红色背景。

### 待催收款项

显示文案：`待催收款项`

统计口径：

```text
payment_status = UNPAID
且已达到催款提醒日期
且当前仍需要催收
```

不是全部未付款运单数量。未到提醒日期的未付款运单不计入今日重点。

点击跳转：

```text
/shipments/payment-reminders?scope=todo
```

视觉建议：橙色。

### 轨迹审批

显示文案：`轨迹审批`

统计口径：

```text
review_status = pending_review
```

点击跳转：

```text
/approvals/tracking-time?status=pending_review
```

视觉建议：紫色。

### 三天内到港

显示文案：`三天内到港`

统计口径：

```text
ETA >= 今天 00:00
ETA < 今天 + 4 天 00:00
ATA 为空
运单未取消
```

即包含今天、明天、后天和第三天内的到港事项。具体日期边界由后端统一计算，前端不自行拼接时间条件。

点击跳转应进入运单或船期页面的“三天内到港”筛选结果。第一版建议继续进入船期监控：

```text
/vessel-schedules?maritimeStatus=arriving_soon
```

视觉建议：蓝色。

### 数字显示

- `0～99` 显示精确数字。
- `100` 及以上可以显示 `99+`。
- 鼠标悬停或辅助说明中显示精确数字。
- 点击卡片必须携带对应筛选条件进入业务页面。
- 卡片不展示装饰折线。

## 待处理事项

### 页面职责

待处理事项展示当前最需要人工处理的事项，不展示所有通知和所有历史提醒。

第一版来源：

- 运单异常
- 催款提醒
- 轨迹审批

后续可以增加：

- 报价跟进
- 快递官网轨迹异常
- 其他明确需要人工处理的业务任务

### 分类切换

顶部使用轻量 Tabs 或分段控制：

```text
全部 / 异常 / 催款 / 审批
```

默认选中“全部”。分类切换只改变当前列表，不改变今日重点数字。

### 展示数量

- 默认最多展示 6～8 条。
- 超出部分通过“查看全部”进入相应业务页面。
- 首页不使用无限滚动和分页表格。
- 没有待办时展示简短空状态，不用大面积插图。

### 排序规则

“全部”列表不只按创建时间排序，建议使用统一优先级：

1. 严重超时、重大异常。
2. 已逾期催款或今天必须催款。
3. 长时间未处理的轨迹审批。
4. 普通待处理异常。
5. 普通催款提醒和普通审批。

同一优先级内按以下顺序：

```text
逾期天数降序
触发时间升序
更新时间升序
```

即越严重、越早产生且越久未处理的事项越靠前。

### 同票事项合并

同一运单存在多个相近提醒时，首页应尽量合并，避免同一个运单连续占据多行。

示例：

```text
DPSECO260718001
3 项待处理：严重超时、轨迹异常、待催款
```

合并只影响首页展示，不修改各业务模块原始记录。

同票存在不同类型事项时，使用最高优先级事项决定列表位置。

### 每条信息

每条事项建议展示：

- 类型与严重程度
- 运单号或分组号
- 客户或关键业务对象
- 简短原因
- 触发时间或逾期时长
- 一个主要跳转动作

首页以查看和跳转为主。催款备注、异常跟进、审批详情等复杂操作进入对应业务页面完成。

不建议在每条记录上同时放置“知道了、已处理、查看、编辑”等多个按钮。

## 近期到港

近期到港用于展示未来三天的重要运输节点，不重复展示完整船期监控页面。

### 日期分组

```text
今天
明天
后天及以后
```

每组内部按 ETA 升序。

### 航次聚合

同一船名航次、目的港和 ETA 下有多票运单时，优先聚合展示：

```text
OOCL TURKIYE / 011W
Felixstowe · ETA 07-20 15:00 · 13 票
```

点击后进入对应航次或运单筛选结果，避免首页重复列出 13 条相同航次记录。

### 展示数量

- 默认展示 5～6 个聚合节点。
- 没有近期到港时显示“未来三天暂无到港”。
- “查看全部”进入船期监控的到港筛选。

### 订阅港

订阅港是近期节点的附加标识，不再单独占据一个完整首页模块。

例如：

```text
Felixstowe · 订阅港
```

港口到港消息是否已读继续由消息系统管理，不影响三天内到港的业务统计。

## 运输概览

运输概览放在主要待办区域下方，视觉权重低于今日重点。

第一版建议四项：

```text
在途
查验中
已到港
本周签收
```

概览用于了解业务规模，不使用异常色大面积强调，也不进入顶部待办。

建议口径：

- 在途：当前运输状态为在途，且未签收、未取消。
- 查验中：当前有效异常为查验。
- 已到港：已经存在 ATA，但 `delivered_time` 为空，且运单未取消。
- 本周签收：`delivered_time` 在本周内。

“已到港”统一采用“已到港但尚未签收”的口径，不累计已经签收的历史到港运单。该指标用于观察已经进入目的端、仍需继续跟进派送或签收的活跃运单。

运输概览只展示数字和名称，不展示静态折线。未来接入真实日序列后，可以增加同比、环比或趋势。

## 首页与顶部铃铛的分工

```text
顶部铃铛 = 有新事项的轻量入口
首页工作台 = 当前最重要事项的优先级视图
业务页面 = 完整查询、处理和历史记录
```

三者必须共用业务状态：

- 在业务页面处理完成后，首页和铃铛同步消失或更新。
- 仅点击“知道了”只消除消息未读状态，不等于业务已处理。
- 真正的待办必须通过跟进、审批、关闭、付款等业务动作完成。
- 首页不重新创建一套独立待办状态。

顶部铃铛不展示全部未付款、全部在途或全部异常历史，只展示新增、紧急或确实需要处理的提醒。

## 数据待修正

以下记录不应作为普通关注运单长期展示：

- 很早以前的 ETA/ETD，但仍标记在途。
- 长期无轨迹更新。
- 状态与 ATA、签收时间等关键字段冲突。
- 已取消或已签收但仍参与在途统计。

首页可以在运输概览下方提供一个低权重入口：

```text
数据待修正 12
```

第一版没有成熟数据质量规则时可以暂不展示，但普通“近期到港”和“关注事项”必须排除明显无效历史记录。

## 数据加载与容错

当前首页依赖多个业务接口。改版后各模块应独立加载，避免任意接口失败导致整个首页清空。

建议结构：

```text
今日重点：独立请求或聚合摘要接口
待处理事项：独立请求
近期到港：独立请求
运输概览：独立请求
```

可以使用后端聚合接口减少请求数量，但聚合服务内部也应允许部分结果失败，并返回模块级错误。

响应建议：

```json
{
  "generatedAt": "2026-07-18 16:30:00",
  "focus": {},
  "todos": { "available": true, "items": [] },
  "arrivals": { "available": true, "items": [] },
  "overview": { "available": false, "error": "统计暂不可用" }
}
```

前端行为：

- 某一模块失败，只在该模块显示轻量错误和重试按钮。
- 已成功加载的其他模块正常展示。
- 刷新时保留已有数据，使用局部加载状态，避免整页闪烁。
- 页面右上角显示最后一次成功更新时间。
- 后续可以每 5～10 分钟静默刷新一次。

## 后端接口建议

可以新增工作台聚合接口：

```text
GET /api/v1/workbench/overview
```

建议查询参数：

```text
todoLimit=8
arrivalLimit=6
timezone=Asia/Shanghai
```

也可以分阶段复用现有接口：

- 运单异常摘要与列表
- 催款列表统计
- 轨迹审批待处理数量
- 海运预警概览

无论采用聚合接口还是多个接口，四项今日重点的统计口径必须由后端统一计算，不能由前端从当前列表数量推断。

## 前端组件建议

当前首页业务集中在一个大型海运预警组件中。改版建议拆分为：

```text
HomeView
  WorkbenchHeader
  WorkbenchFocusMetrics
  WorkbenchTodoList
  WorkbenchArrivalTimeline
  WorkbenchTransportOverview
  WorkbenchDataQualityLink（后续）
```

组件按业务职责拆分，避免继续把催款、报价、审批等逻辑加入 `MaritimeAlertsPanel`。

页面容器保持安静、紧凑的运营工具风格：

- 不使用大面积 Hero。
- 不嵌套多层卡片。
- 四项重点指标保持固定尺寸。
- 待办列表和到港列表使用稳定行高。
- 颜色只表达类型和严重程度，不作为纯装饰。
- 数字使用等宽数字特性，避免刷新时跳动。

## 跳转参数

建议统一首页跳转参数，确保点击后看到与首页数字一致的结果：

| 首页入口 | 目标页面 | 建议参数 |
| --- | --- | --- |
| 待处理异常 | `/shipment-exceptions` | `status=open` |
| 待催收款项 | `/shipments/payment-reminders` | `scope=todo` |
| 轨迹审批 | `/approvals/tracking-time` | `status=pending_review` |
| 三天内到港 | `/vessel-schedules` | `maritimeStatus=arriving_soon` |
| 在途 | `/shipments` | `statusCode=IN_TRANSIT` |
| 查验中 | `/shipments` 或异常页 | `exceptionCode=INSPECTION` |
| 本周签收 | `/shipments` | `deliveredFrom` / `deliveredTo`（本周周一～周日） |
| 已到港未签收 | `/shipments` | `hasAta=true&notDelivered=true` |

业务页面必须支持这些参数并保持筛选状态可见。

## 第一阶段实施顺序

### 阶段一：清理首屏

1. 替换大 Hero 为紧凑页头。
2. 删除首页快捷导航。
3. 删除原七张海运统计卡和静态折线。
4. 增加四项今日重点。
5. 保留现有接口，先完成统计口径与跳转对齐。

### 阶段二：重组主体

1. 合并异常、催款和审批为待处理事项。
2. 增加分类切换和统一优先级。
3. 将海运预警、关注运单和订阅港重组为近期到港。
4. 对同航次多票进行聚合。

### 阶段三：概览与稳定性

1. 增加运输概览。
2. 各模块独立加载和失败重试。
3. 增加静默刷新和最后更新时间。
4. 增加数据待修正入口。

### 阶段四：后续扩展

1. 报价跟进进入今日待办来源。
2. 快递官网轨迹异常进入今日待办来源。
3. 根据实际使用情况增加个人化配置。
4. 有真实历史数据后增加趋势指标。

## 验收标准

1. 1280×720 桌面视口首屏可看到完整今日重点和待处理区域主要内容。
2. 今日重点固定为待处理异常、待催收款项、轨迹审批、三天内到港。
3. 四项数字与目标业务页面筛选结果口径一致。
4. 首页不再重复展示快捷导航和原七张海运统计卡。
5. 待处理事项默认不超过 8 条，并按统一优先级排序。
6. 同票相近事项不会连续重复占据多行。
7. 近期到港按日期和航次聚合，默认不超过 6 条。
8. 首页处理状态与顶部铃铛、业务页面保持一致。
9. 单个模块接口失败时，其他模块仍可正常展示。
10. 历史明显无效的在途记录不进入普通近期关注列表。
11. 页面不展示伪趋势或装饰性折线。
12. 移动端无横向溢出，四项重点可使用两列布局。

## 已确认口径

运输概览中的“已到港”采用：

```text
ATA 已存在
且 delivered_time 为空
```
