# 运输时效统计管理设计方案

## 背景

当前 `统计管理` 页面已经具备运单总数、状态分布、渠道占比、承运商占比和基于 `ATD` / `ATA` 的时效基准线预览，但还没有真正承接业务分析场景。

随着内部轨迹时间字段回写能力落地，运单会逐步拥有以下关键时间：

- `ETD`：预计离港时间
- `ATD`：实际离港时间
- `ETA`：预计到港时间
- `ATA`：实际到港时间
- `expected_delivery_time`：预计送仓时间
- `signed_time`：签收时间

这些字段可以支撑统计管理从“数量分布”升级为“运输时效分析中心”，用于回答以下问题：

- 某个渠道整体运输时效如何？
- 某个承运商在某个渠道的运输时效如何？
- 某个客户的运输时效和稳定性如何？
- 哪些渠道、承运商、客户存在明显延误？
- ETA、预计送仓时间的准确性如何？

## 菜单归属

功能放在现有菜单：

```text
数据中心 / 统计管理
```

原因：

- 该功能关注聚合分析，不是单票运单操作。
- 统计对象包括渠道、承运商、客户、目的港、时间周期等多个维度。
- 现有统计管理已经是数据分析入口，适合承载运输时效分析。

不建议放在 `运单中心`。运单中心更适合单票运单管理、轨迹审批和异常跟进。

## 设计目标

1. 基于 `ETD`、`ATD`、`ETA`、`ATA`、`expected_delivery_time`、`signed_time` 计算运输时效。
2. 支持按渠道、承运商、客户、承运商 + 渠道等维度聚合。
3. 支持时间范围筛选，并允许选择统计口径时间。
4. 展示平均值、P50、P90、最大值、异常率、完成率等关键指标。
5. 从聚合结果可以下钻到运单明细，便于排查慢单。
6. 首版优先做可用的表格和 KPI，再逐步补充趋势图和分布图。

## 核心指标

### 时效类指标

| 指标 | 计算方式 | 说明 |
| --- | --- | --- |
| 开船偏差 | `ATD - ETD` | 实际开船比预计开船早/晚几天 |
| ETA 偏差 | `ATA - ETA` | 实际到港比预计到港早/晚几天 |
| 海运时效 | `ATA - ATD` | 实际离港到实际到港 |
| 到港后送仓时效 | `signed_time - ATA` | 实际到港到签收 |
| 全程实际时效 | `signed_time - ATD` | 实际离港到签收 |
| 计划全程时效 | `expected_delivery_time - ETD` | 预计离港到预计送仓 |
| 预计送仓偏差 | `signed_time - expected_delivery_time` | 签收比预计送仓早/晚几天 |

首版重点指标：

```text
海运时效 = ATA - ATD
到港后送仓时效 = signed_time - ATA
全程实际时效 = signed_time - ATD
预计送仓偏差 = signed_time - expected_delivery_time
```

### 完成率和异常率

| 指标 | 计算方式 | 说明 |
| --- | --- | --- |
| 签收完成率 | 有 `signed_time` 的票数 / 总票数 | 渠道闭环情况 |
| 到港完成率 | 有 `ATA` 的票数 / 总票数 | 到港数据完整度 |
| 海运时效可计算率 | 同时有 `ATD` 和 `ATA` 的票数 / 总票数 | 海运段统计样本覆盖率 |
| 全程时效可计算率 | 同时有 `ATD` 和 `signed_time` 的票数 / 总票数 | 全程统计样本覆盖率 |
| 异常率 | 异常票数 / 可计算票数 | 时效稳定性 |

## 统计维度

首版筛选维度：

```text

```

后续扩展维度：

```text

国家 / 区域
业务员 / 操作员

```

### 时间范围口径

统计页需要支持选择时间口径：

| 口径 | 字段 | 使用场景 |
| --- | --- | --- |
| 按实际离港 | `ATD` | 分析出运批次的运输表现，推荐默认 |
| 按实际到港 | `ATA` | 分析目的港处理和送仓表现 |
| 按签收时间 | `signed_time` | 分析已闭环订单表现 |
| 按创建时间 | `created_time` | 兼容历史运单和数据看板 |

默认使用：

```text
按 ATD 统计
```

原因是运输时效通常以实际出运批次作为分析起点，更适合比较渠道、承运商和客户。

## 渠道时效

按 `channel_code` 聚合。

字段：

| 字段 | 说明 |
| --- | --- |
| 渠道 | `channel_code` |
| 总票数 | 筛选范围内票数 |
| 已签收票数 | 有 `signed_time` 的票数 |
| 签收完成率 | 已签收票数 / 总票数 |
| 平均海运时效 | 平均 `ATA - ATD` |
| 平均到港后送仓时效 | 平均 `signed_time - ATA` |
| 平均全程实际时效 | 平均 `signed_time - ATD` |
| P50 全程时效 | 全程时效中位数 |
| P90 全程时效 | 90 分位全程时效 |
| 平均预计送仓偏差 | 平均 `signed_time - expected_delivery_time` |
| 异常票数 | 命中异常规则的票数 |
| 异常率 | 异常票数 / 可计算票数 |

用途：

```text
判断某个渠道整体是否稳定
比较不同渠道的平均时效和尾部慢单
识别渠道时效恶化
```

## 异常规则

首版可以先内置基础异常规则：

| 异常 | 判断方式 |
| --- | --- |
| 晚开船 | `ATD - ETD > 0` |
| 晚到港 | `ATA - ETA > 0` |
| 送仓晚于预计 | `signed_time - expected_delivery_time > 0` |
| 到港后送仓过慢 | `signed_time - ATA > 7` |
| 全程超时 | `signed_time - ATD > 渠道标准时效 + N 天` |

其中 `全程超时` 依赖渠道标准时效。首版如果还没有渠道标准配置，可以先用固定阈值或只展示偏差，不直接判异常。

推荐首版先启用：

```text
晚到港
送仓晚于预计
到港后送仓过慢
```

## 数据计算口径

### 日期差

首版按日期计算，不比较时分秒：

```text
days = date(end_time) - date(start_time)
```

例如：

```text
2026-06-15 09:30
2026-06-16 01:00
```

计算为 1 天。

### 样本过滤

某个指标只统计字段完整的运单：

```text
海运时效：必须同时有 ATD 和 ATA
到港后送仓时效：必须同时有 ATA 和 signed_time
全程实际时效：必须同时有 ATD 和 signed_time
预计送仓偏差：必须同时有 expected_delivery_time 和 signed_time
```

页面上需要展示样本数，避免平均值被误解。

示例：

```text
平均全程实际时效：31.2 天
样本：128 / 203 单
```

### 分位数

P50 / P90 只基于可计算样本。

排序规则：

```text
取所有可计算天数升序排列
P50 = 中位数
P90 = 90 分位位置的值
```

首版也可以先只做平均值，P50 / P90 作为第二阶段。


## 首版落地范围

建议第一版实现：

1. 顶部筛选：时间范围、时间口径、渠道、承运商、客户。
2. KPI：总票数、已签收票数、签收完成率、平均海运时效、平均全程实际时效、平均预计送仓偏差。
3. 聚合表：按渠道、按承运商 + 渠道、按客户。
4. 明细表：展示关键时间字段和计算出的时效。
5. 导出 CSV。

暂不强制实现：

```text
趋势图
箱线图
P50 / P90
渠道标准时效配置
同比环比
```

## 后续扩展

1. 增加渠道标准时效配置，用于更准确判断全程超时。
2. 增加趋势图，查看渠道或承运商时效是否变差。
3. 增加 P50、P90、P95，关注尾部慢单。
4. 增加客户维度的时效报告导出。
5. 将异常票推送到首页通知或运单异常任务。
6. 支持按 Amazon FC、目的港、船名航次继续下钻。
